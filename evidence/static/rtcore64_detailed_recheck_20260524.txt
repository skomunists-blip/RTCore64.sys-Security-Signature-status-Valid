F1) FUN_00011384: slot-table full -> success without rollback
====================================================================
Status: CONFIRMED (static logic)
Confidence: High

Why:
- In FUN_00011384, after successful MmMapIoSpace:
  - table scan loop checks up to 0x100 slots (DAT_00014370)
  - if no empty slot, flow jumps to common success path
  - uVar3 is set to STATUS_SUCCESS (0) regardless
- No MmUnmapIoSpace rollback exists in this path.

Key evidence:
- rtcore_deep_issue_hunt.txt
  - 000113db CALL MmMapIoSpace
  - 00011402 CMP EDX,0x100
  - 0001140a JMP 0x00011421
  - 00011421 XOR EAX,EAX (success)
- rtcore_comprehensive_audit.txt decompile of FUN_00011384

Impact:
- leaked/untracked kernel mapping candidate
- possible resource exhaustion / reliability degradation
- false-success semantic (caller sees success despite lost bookkeeping)

F3) FUN_00011150 lifetime path (object reference + handle cleanup)
====================================================================
Status: PROBABLE (partially confirmed, one previous claim needs correction)
Confidence: Medium-High

- ObDereferenceObject is not found in imports/calls.
- Referenced object is not required for ZwMapViewOfSection path itself.

What needed correction from earlier wording:
- Earlier summary said "if ObReferenceObjectByHandle fails, opened section handle is returned without ZwClose."
- In decompiled flow, ObReferenceObjectByHandle failure happens before later ZwClose points and appears to return directly.
- That still indicates handle-lifetime scrutiny is needed, but the exact leak condition should be reported as "likely" unless revalidated in raw disassembly control-flow with all exceptional exits.

Key evidence:
- rtcore_deep_issue_hunt.txt decompile/disasm for FUN_00011150
- rtcore_comprehensive_audit.txt imports + callsites

Recommended report wording:
- "Likely reference-lifetime issue: ObReferenceObjectByHandle is used without corresponding ObDereferenceObject visibility in this module."
